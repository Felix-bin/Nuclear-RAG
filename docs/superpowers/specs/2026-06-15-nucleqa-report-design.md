# NucleQA 课程设计报告完善设计

## 1. 目标

基于仓库当前实际内容与既有 `应用软件架构课程设计报告模板.docx`，产出一份可直接提交的《应用软件架构课程设计报告》。成品以课程评分点为主线，同时保证论述完整、结构清晰、风格正式，避免写成仅罗列截图的填空文档。

本次工作不改动项目业务代码，不扩展项目功能，仅围绕课程报告进行内容整理、图示生成、证据截图、版式填充与文档校验。

## 2. 依据材料

报告内容仅基于以下真实材料组织，不虚构未落地能力：

- 仓库根目录 `README.md`
- `docs/latest/intro/project-overview.md`
- `docs/latest/intro/knowledge-base.md`
- `docs/latest/intro/quick-start.md`
- `pyproject.toml`
- `package.json`
- 当前源码目录结构（`src/`、`server/`、`web/`）
- 模板内已有题目说明与课程要求

若模板中的个别表述与仓库真实实现不一致，以仓库真实实现为准，并在报告中采用更准确的项目名称、技术栈和模块职责。

## 3. 报告定位

采用“评分点驱动 + 架构分析强化版”的写法：

- 保留模板既定章节顺序，确保逐项覆盖评分点
- 每节均使用项目真实实现作为支撑
- 重点强化技术架构、业务架构、核心功能设计与接口使用分析
- 行文风格保持学术化、完整化，避免口语化与流水账式描述

整体定位为“课程报告结构为主，软件架构分析为核心”。

## 4. 报告结构设计

### 4.1 功能实现展示

该部分按“功能截图 + 场景说明 + 价值说明”的模式组织，计划选择 5 个功能点，兼顾评分要求与项目代表性：

1. 智能问答与引用溯源
2. 知识库创建、文档上传与解析入库
3. 知识图谱/LightRAG 图谱可视化
4. 对话历史、反馈与任务流转
5. 部门、用户与知识库权限管理

其中“必须包含 1 个业务表/主从关系场景”的评分要求，落在以下真实业务关系之一：

- 知识库与文档的主从关系
- 部门、用户、知识库授权之间的关联关系

报告不会刻意伪造传统“订单-订单项”式案例，而是直接用本项目已有业务实体关系完成评分点覆盖。

### 4.2 技术角度的技术架构图

该部分只呈现技术层次与基础设施，不引入业务对象。计划至少放入 2 张图：

- 系统总体技术架构图
- 部署/运行时组件图

图中重点体现：

- Vue 3 + Vite 前端
- FastAPI 后端服务
- LangGraph 智能体编排
- Milvus 向量检索
- Neo4j 图数据库
- PostgreSQL 关系数据存储
- MinIO 文件对象存储
- Docker Compose 容器化运行方式

### 4.3 业务角度的业务架构图

该部分强调“业务如何被架构支撑”，而不是功能清单。计划至少放入 2 张图：

- 面向用户请求流的业务交互图
- 面向知识生命周期的业务处理图

图中将包含：

- 用户角色
- 前端业务模块
- 问答服务模块
- 知识库管理模块
- 文档解析与入库模块
- 检索生成模块
- 图谱增强模块
- 审计与反馈模块

并明确模块间输入输出、接口调用与数据落点。

### 4.4 核心架构设计

该部分围绕 2 个最能体现架构特征的具体功能点深入展开：

#### 功能点 A：知识库文档上传 - 解析 - 入库

覆盖内容：

- 前端上传与任务触发入口
- 后端知识库/文档 API
- 文档解析与异步任务处理
- 文本分块、向量化、图谱构建
- 元数据、对象存储、向量库、图数据库的协同

#### 功能点 B：RAG 问答 - 检索 - 生成 - 溯源

覆盖内容：

- 前端问答页面与流式交互
- 后端聊天接口暴露方式
- LangGraph/检索流程组织
- 知识库召回、图谱增强、生成回答
- 引用来源、权限与合规约束

模板要求的前端层、后台架构、控制层接口、业务层、数据操作层，都将嵌入这 2 个功能点中展开，而不是分裂成互不关联的碎片描述。

### 4.5 接口技术的使用

选择 2 项最贴合项目实际的技术能力：

1. 大模型 API / 聊天模型接入
2. Neo4j 或 Milvus 相关接口能力

如果在取证过程中发现项目中对某类接口呈现得更完整，也可替换为：

- 文档解析/OCR 接口
- MCP 服务接口
- MinIO 文件存储接口

选择标准是“代码证据充分、能清楚展示配置与调用链路”。

## 5. 图像与截图策略

### 5.1 使用 AI 生成的图

以下内容使用 `gpt-image-2` 生成：

- 技术架构图
- 业务架构图
- 部署拓扑图
- 数据流/处理流示意图

统一风格要求：

- 科研学术风格
- 白底或浅底
- 克制用色，蓝灰为主
- 信息层次清晰
- 适合嵌入 Word 报告，不做宣传海报风

### 5.2 使用真实项目截图

以下内容必须优先使用真实截图：

- 系统功能页面
- 代码片段
- 配置文件
- 目录结构
- 接口定义

原则是“图示负责抽象表达，截图负责真实性证明”。

## 6. 文档写作与排版原则

- 用中文完成所有非代码内容
- 标题层级清晰，段落完整，不写成短句堆砌
- 每个功能点尽量遵循“目标 - 实现 - 效果”叙述方式
- 每张图都配图号与说明文字
- 每段截图说明都明确其在系统中的角色
- 不额外添加模板未要求的大章节，但会在各节内部通过过渡文字增强完整性

## 7. 实施步骤

1. 提取模板结构并确认写法方向  
   验证：章节与评分点全部被覆盖

2. 盘点仓库中可用于报告的真实证据  
   验证：功能、代码、接口、配置、目录均有对应材料

3. 设计并生成技术图与业务图  
   验证：图示内容与真实架构一致，风格统一

4. 生成或采集功能截图与代码截图  
   验证：截图能支撑模板要求的 4-6 个功能点与 2 个核心功能分析

5. 填充 DOCX 模板正文与图片  
   验证：所有占位段落被替换，论述连续，图文对应

6. 渲染并检查 DOCX 页面效果  
   验证：无明显分页错乱、图文重叠、标题断裂、表格溢出

7. 输出最终 DOCX 与素材  
   验证：成品可直接提交，素材路径清晰

## 8. 风险与处理

### 8.1 模板与仓库实现存在轻微不一致

例如模板示例中偏传统 Java 多层架构，但本项目为 Python + FastAPI + Vue + LangGraph。处理方式是沿用模板评价逻辑，但将“Controller/Service/Dao”映射到本项目真实的路由层、服务层、仓储/存储层。

### 8.2 某些功能未必有现成漂亮截图

如果个别页面截图表现力不足，则以真实截图为主，必要时辅以 AI 生成的辅助架构图解释其位置和流程，但不伪造功能界面。

### 8.3 配图与实际实现偏离

所有 AI 图示都必须在生成前先根据仓库结构列出要素清单，生成后再对照校正，确保不出现模板化乱画。

## 9. 完成标准

满足以下条件即可视为本任务完成：

- 已基于模板生成完整课程报告
- 已覆盖模板全部核心评分项
- 已体现项目真实技术架构与业务架构
- 已包含学术风格图示、真实功能截图、真实代码截图
- 已完成 DOCX 渲染检查，版式可读

## 10. 当前结论

后续执行时，不追求把报告写成脱离模板的独立论文，而是在模板框架内做到：

- 内容完整
- 结构漂亮
- 论述扎实
- 图文可信
- 架构分析有深度

这将作为后续文档补全与配图生成的唯一执行基线。
